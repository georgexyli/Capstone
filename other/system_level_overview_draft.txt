System-Level Overview (author: Y. Fikry)

The DeFi AI Agent is architected as a gated execution pipeline that transforms natural language user input into validated, confirmed, and broadcast blockchain transactions. The system follows a strict sequential flow in which each stage must complete successfully before the next stage begins, and no on-chain action can occur without explicit user approval. Figure X presents the system block diagram. The final system follows the nine-module architecture we laid out in the Implementation Plan, with all modules built and working in the delivered prototype. We did not deviate from the original design; the pipeline stages, their ordering, and the interfaces between them are the same as what was proposed.

At the highest level, the system comprises three layers. The messaging interface layer handles user-facing communication through a Telegram bot, receiving commands and returning previews, confirmations, and receipts. The core execution pipeline contains the business logic that parses intent, obtains quotes, enforces safety checks, and orchestrates transaction flow. The blockchain interaction layer manages direct communication with Ethereum nodes via JSON-RPC, including contract reads, transaction signing, and broadcast.

The pipeline processes each swap request through seven stages: (1) intent parsing, where the user's natural language message is converted into structured fields by an LLM; (2) routing and quote generation, where the Uniswap V3 QuoterV2 contract is queried for the best price across fee tiers; (3) human confirmation, where a preview is shown and execution halts until the user explicitly approves; (4) pre-execution simulation, where balance, allowance, gas, and slippage are verified before signing; (5) transaction construction and signing, where calldata is built and signed locally using the user's private key; (6) broadcast and receipt, where the signed transaction is submitted and confirmed on-chain; and (7) post-trade rendering, where the actual output is decoded from event logs and a detailed receipt is returned to the user. Each stage is described in detail in Section 4.2.
